# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Status

"Maze Dodge" — a small, self-contained 2D browser game. No git repo, no build tooling, no dependencies; plain HTML/CSS/JS with the Canvas API. Sibling project to `../2D game` ("Circle Survivor"), sharing its lightweight conventions but with unrelated gameplay logic.

## Running it

Open `index.html` directly in a browser, or serve the directory with `python3 -m http.server` and visit `localhost:8000`. A project `run` skill (`.claude/skills/run/SKILL.md`) documents a headless-Chrome/CDP recipe for driving and verifying the game end-to-end — read it before improvising a new browser-testing approach, and note its warning about launching Chrome with the target URL as the startup argument (not `about:blank` + navigate), since the latter can silently leave the tab backgrounded and `requestAnimationFrame` fully suspended.

## Architecture

- `index.html` — canvas element (800×600, no scrolling/camera — each maze fills the visible grid) plus DOM overlay divs (start/username/pause/leaderboard screens, HUD, a reusable banner). Overlays are toggled via the `.hidden` CSS class.
- `style.css` — layout and overlay styling.
- `game.js` — all game logic, in one file:
  - A `Phase` state machine (`start` → `menu` → `transform_in` → `playing` ⇄ `paused` → `transform_out` → back to `menu`) drives `update()`/`render()`. `start` is only ever visited once per session; every subsequent run loops back through `menu`.
  - The story: a traveler crossing a mountain range village by village, using a "become small and quick" trick to slip through tight places — a lighter journey framing (not a chase). Hazards (patrol/circular/bounce shapes) are the only visible obstacles — no enemy sprites, by design, just re-flavored as lanterns/beacons/wandering lights rather than pursuers. `LEVELS` holds 8 entries: 7 hand-authored World-1 maze levels named after villages along the range (Stonebridge Hollow → Skyreach Cairn, plus **Wyrmwind Col** as level 7), each an ASCII grid (`#`/`.`/`S`/`E`, 20×15 cells at `CELL_SIZE`=40) plus a `hazards` array, a one-line `flavorText`, a `collectibles` array, and (level 7 only) a `dashOrbs` array — plus World 2's single runner-mode level (index 7, "The Long Drop", see below). Levels 1–6 were generated with a recursive-backtracker maze algorithm and verified solvable via BFS before being hand-tuned with hazard placements — treat the grids as fixed data, not something to hand-edit casually without re-checking solvability. Level 7 (Wyrmwind Col) is a hand-authored single 1-wide winding corridor (see "Dash orbs" below).
  - **`WORLD_PATH`/`LEVELS` positional coupling is an undocumented-in-code invariant.** Nothing enforces it structurally, but `renderMenu()`'s node loop, `nodeIndexAt()`, and `enterLevel(idx)` all assume `WORLD_PATH[i]` is the map node for `LEVELS[i]`. Adding a level means appending to *both* arrays at the same index — never insert in the middle or reorder one without the other.
  - **Two worlds, one flat `LEVELS`/`progress`.** `WORLDS` (near `WORLD_PATH`) is a thin view layer: each entry is `{name, theme, startIndex, path}`, where `path` is that world's own node array (`WORLD_PATH` for World 1, the single-node `CAVERN_PATH` for World 2) and `startIndex` is its offset into the still-flat `LEVELS`/`progress.*` arrays — `worldIndexForLevel(levelIdx)` maps a flat index back to a world. **This is deliberately not a deeper restructuring**: `ensureProgressShape()` already generically pads `gemsCollected`/`bestTimes`/`completedLevels` to `LEVELS.length` and re-derives `unlockedIndex` from `completedLevels`, so appending World 2 as `LEVELS[7]` needed zero changes there — world-unlocking is a pure byproduct of the existing flat `unlockedIndex` gate, not a new mechanism. `nodeIndexAt(mx, my, world)` now takes a world and returns a flat level index (`world.startIndex + i`). `viewedWorldIndex` (module state) tracks which world's map is currently displayed — it follows the player's frontier by default, advances automatically the moment `finishLevel()` unlocks a new world, but otherwise stays put so browsing an earlier, fully-cleared world (e.g. to mop up missed gems) doesn't get yanked back. Small on-canvas arrow hit-targets (`worldSwitchHitTest`/`switchViewedWorld`, drawn by `drawWorldSwitchArrows`) let the player switch manually, gated by `canViewWorld()` (a world is viewable once its `startIndex <= progress.unlockedIndex`) — both go through `canvasCoords(e)` like every other click handler, never raw `clientX/clientY`.
  - **Dynamic (hazard) solvability matters, not just wall solvability.** A patrol/circular/bounce hazard placed inside a corridor that's exactly 1 cell wide with no lateral escape can make a level provably uncrossable regardless of its speed or patrol length (a continuously-moving hazard confined to a range the player must fully cross guarantees their positions coincide at some instant — an intermediate-value argument, not just "hard"). This bit two levels in practice (level 2's entry corridor and a level 4 corridor) until fixed by either opening a wall cell to create a real bypass route, or relocating the hazard into a room with genuine lateral room. When adding or moving hazards, verify a route exists that avoids the hazard's full reachable range (with margin = player radius + hazard radius), or that its reachable cells are mostly *not* 1-wide tubes — don't just eyeball it. `tools/solvability_checker.py` (stdlib-only Python, no deps) reimplements the relevant subset of `parseLevel`/`createHazard`/`updateHazard` to do this: run `python3 tools/solvability_checker.py` for a per-level, per-hazard bypass/tube report (it already knows Level 7's two hazards are intentionally dash-gated via its `DASH_GATED_HAZARDS` allowlist, so a clean run doesn't mean "no dash-gated hazards," it means none of them are *newly* broken). It also exposes `candidate_collectible_cells()` (off-route + hazard-free cell search, used to place every existing gem) and `dash_reach_ok()` (wall-clearance + sufficient-reach check for a dash past a hazard) for adding new content. Keep it in sync with `game.js` if the hazard/movement math ever changes there.
  - **`reactive` hazards need no extra solvability check.** The reachable-range argument above is spatial, not speed-dependent — a hazard's reachable range (segment/circle/rect) is fixed geometry regardless of how fast it moves; going faster only changes arrival timing, which the check never modeled. So a `reactive` flag on a hazard def (see below) is safe under the existing check as-is.
  - **Pausing**: `Escape` toggles `Phase.PLAYING` ⇄ `Phase.PAUSED` (`pauseGame`/`resumeGame`), showing/hiding `#pauseScreen`. `update()` simply has no branch for `PAUSED`, so gameplay (movement, hazards, collisions, the level timer) freezes automatically; `render()` treats `PAUSED` the same as `PLAYING` so the frozen scene stays visible under the overlay. "Exit to Map" calls the existing `returnToMenu(null)` directly — a voluntary exit never touches `progress`/`saveProgress`, unlike death or level-complete.
  - **World map (`Phase.MENU`)**: a static, once-built mountain-range vista (`buildMapBackground()` → offscreen `mapBackgroundCanvas`, blitted each frame via `drawImage`) — sky/sun/clouds, 4 seeded ridgeline layers with snow-caps (`generateRidgeline`/`drawRidgeLayer`), a river (plus a live `drawRiverFlow()` overlay for motion), a valley floor with seeded pine trees, and the trail all baked in once at startup since none of it changes at runtime. `WORLD_PATH` is a fixed array of node coordinates; `renderMenu()` draws each node as a cave entrance set into a rock mound (`drawVillageNode`) with locked (rubble-blocked, 🔒)/unlocked (dark mouth with a warm glow)/completed (gold-toned + green glow ring)/hover states — hit-testing (`nodeIndexAt`/`NODE_RADIUS`) is independent of the node art and must stay in sync with `WORLD_PATH`. A completed node also gets its best-time drawn beneath it when hovered, and the lifetime gem total is drawn top-right. Progress persists via `localStorage` under `mazeDodgeProgress` (`loadProgress`/`saveProgress`); `ensureProgressShape()` backfills newly-added fields (and pads arrays / recomputes `unlockedIndex`) onto older saves rather than discarding them. Clicking an unlocked node calls `enterLevel(index)`.
  - **Transformation**: entering/leaving a level plays a short tween (`TRANSFORM_IN`/`TRANSFORM_OUT`, `TRANSFORM_TIME`) blending the animated human silhouette into the small circular in-level form (`drawCharacter`) — purely cosmetic, not a separate animation system.
  - **In-level mechanics**: `updatePlayerMovement` does axis-separated movement against `isWallAt`/`circleCellOverlap` (wall-sliding, no corner-sticking) via a standalone `tryAxis(axis, delta)` helper (also reused by the dash mechanic), and also tracks `player.facingX/facingY/isMoving/animPhase` purely for animation. Hazards (`createHazard`/`updateHazard`) come in three types — `patrol` (ping-pongs between two points), `circular` (orbits a center), `bounce` (reflects off its own bounds rect) — and never collide with walls, so their zones must be placed inside open floor. Hazard speed scales via `speedMul * (HAZARD_SPEED_BASE + levelIndex * HAZARD_SPEED_LEVEL_INCREMENT)`, further boosted by `effectiveHazardSpeed()` when a `reactive: { triggerDist, boostMul }` field is present and the player is within `triggerDist`.
  - **Collectibles**: `collectibles: [{x,y}]` per level (empty on level 7 — a 1-wide corridor has no side branch to hide an optional pickup in). Whether a given gem is taken is read straight from `progress.gemsCollected[levelIndex][gemIndex]` (not local/runtime state) and is never reset once true — a gem, once collected, stays collected permanently across respawns, level failures, and future re-entries alike (`checkCollectiblePickups()` saves immediately on pickup, not batched to level-complete, so it banks even if the attempt is later failed). Placement must satisfy: off the maze's shortest S→E path (a genuine detour), fully outside every hazard's danger zone (both the pickup cell and the corridor leading to it — unlike the exit, an optional pickup should never require dodging a hazard), *and* fully outside the exit's own trigger radius (`exitZone.radius + PLAYER_RADIUS`, matching `checkExitReached()`'s `circlesOverlap` check) — a gem in a dead-end pocket reachable only by first passing through the exit trigger is worse than unsafe, it's flatly impossible, since `checkExitReached()` fires every frame and ends the run before a detour is possible (this shipped as a real bug on level 1 — a gem in a dead-end directly behind the gate, only reachable by walking through the exit cell itself — before being caught and fixed). `tools/solvability_checker.py`'s `candidate_collectible_cells()`/`exit_trigger_cells()` check both hazard danger zones and the exit trigger together now; re-run it after placing any new collectible.
  - **Near-miss feedback**: `checkNearMiss()` (edge-triggered per-hazard via a `nearMissArmed` flag, so it fires once per pass rather than every frame within the band) drives a short expanding/fading ring at the player (`drawNearMissFlash()`) — pure Canvas, no assets, no camera/shake system (this game has neither).
  - **Best-time**: `levelTimer` accumulates only inside `update()`'s `PLAYING` branch (so it's naturally paused-and-resumed for free, and doesn't count the `TRANSFORM_IN` tween), resets on `enterLevel()` but *not* on `respawnInLevel()` (a death mid-attempt doesn't restart the clock). Captured into `progress.bestTimes[levelIndex]` in `checkExitReached()` when it's an improvement, surfaced in the HUD (`#timeLabel`/`#bestLabel`) and on the world map.
  - **Dash orbs (level 7, "Wyrmwind Col")**: this level's grid is a single 1-wide winding corridor with **zero branches** — `parseLevel`/`isWallAt`/collision code needed no changes for this, since none of it assumes maze topology, only "is this cell a wall." A direct structural consequence: on a level with no side branches at all, *any* hazard is either dash-gated or flatly impossible (there's no way to carve a fair dodge pocket without creating an unintended shortcut between adjacent parallel runs) — this level's two hazards are both intentionally dash-gated, not an oversight. `dashOrbs: [{x,y}]` are triggered two ways: clicking one (canvas `click` listener's `Phase.PLAYING` branch, gated on both click-proximity to the orb and player-proximity via `DASH_ORB_PROXIMITY_RADIUS`), or pressing **Space** (`triggerNearestDashOrb()`, keydown handler's `phase === Phase.PLAYING` branch — no aiming needed, just triggers whichever unused orb is nearest within `DASH_ORB_PROXIMITY_RADIUS`, so you don't need to take a hand off WASD). Both call `triggerDashOrb()`: a fixed-duration (`DASH_DURATION`) burst along the player's *current facing direction* (not an authored target point — the corridor is 1-wide, so facing is inherently "along the path" at any orb), reusing `tryAxis` for wall collision (so a dash is a fast sprint, not a teleport — it still can't clip through a wall) and the existing `invulnTimer` field for its invulnerability window. Orbs are consumed **per-attempt** (rebuilt fresh in both `enterLevel()` and `respawnInLevel()`) rather than permanently — a permanently-spent orb would soft-lock a retry against the hazard it exists to bypass. Solvability reasoning for any future dash-gated hazard: the corridor up to and including the orb must pass the *ordinary* hazard-safety check with zero exceptions; only the segment immediately past the orb is allowed to fail it, and that segment instead needs (a) wall-clearance — the dash's full-length straight-line path stays on floor cells — and (b) sufficient reach — that length clears the hazard's danger zone and lands somewhere the *next* segment is ordinary-safe from.
  - **World 2 — "The Deep Cavern" runner level (`LEVELS[7]`, `mode: 'runner'`).** A fundamentally different genre from the top-down maze engine: side-view, player fixed at `RUNNER_PLAYER_X`, spikes scroll in from the right as `runnerScroll` accumulates (`+= RUNNER_SCROLL_SPEED * dt`, the game's only "camera"/scroll concept — confined entirely to obstacle-screen-position math, nothing else gains one), jump height set by how long Space/↑ is held (charge while grounded+holding, launch on release: `velocity = RUNNER_MIN_JUMP_VEL + (RUNNER_MAX_JUMP_VEL-RUNNER_MIN_JUMP_VEL) * (chargeTime/RUNNER_MAX_CHARGE_TIME)`, then simple projectile motion under `RUNNER_GRAVITY`). A level def's `mode: 'runner'` flag is checked at exactly four dispatch points — `enterLevel()`/`respawnInLevel()` (branch to `setupRunnerLevel()` instead of `parseLevel()`), `update()` (branch to `updateRunner(dt)` instead of the maze movement/hazard/collectible/exit chain), and `render()` (branch to `renderRunnerScene()` instead of `renderMaze()` — this is a real correctness fix, not cosmetic, since both `PLAYING` and `TRANSFORM_IN` branches used to call `renderMaze()` unconditionally, which would read stale/undefined `walls`/`exitZone` for a runner level). `loseLife()` and `finishLevel()` were extracted from `checkHazardCollisions`/`checkExitReached`'s bodies so both the maze path and the new `checkRunnerCollisions`/`checkRunnerFinish` share one life-loss/level-complete implementation rather than duplicating it. Pause/resume needed zero changes — `update()` already has no `PAUSED` branch, so gameplay freezes regardless of mode. A hit **restarts the whole gauntlet from `runnerScroll = 0`**, not a checkpoint — consistent with maze respawn always resetting to the parsed `S` cell.
    Obstacles (`gauntlet: {length, obstacles: [{gaugeX, ground: {height}, ceiling?: {height}}]}`) use simple AABB overlap against the player's fixed-width bounding box; a `ceiling` field is only ever paired with a *small* `ground.height`, turning an easy-looking short hop into a precisely-timed one (too little height hits the ground spike, too much hits the ceiling spike above it) — this is the intentional "narrow gap" pattern, not every obstacle. **Solvability here isn't grid-BFS** — `tools/solvability_checker.py`'s regex requires a `grid:` key and silently skips this level. `tools/runner_solvability_checker.py` re-simulates the exact same charge-jump physics model (**keep both files' `RUNNER_*` constants in sync**) to prove, per obstacle independently (the player chooses release timing freely, so obstacles don't interact), that some hold duration clears it, plus a spacing check between consecutive obstacles. That static model is necessarily a simplified proof of existence, not a guarantee of real in-game feel — the CDP playtest (`.claude/skills/run/SKILL.md`) is what actually exercises `updateRunner`/`checkRunnerCollisions` end-to-end; a real bug (`RUNNER_PLAYER_HEIGHT` was referenced in `checkRunnerCollisions` but never defined, silently killing the `requestAnimationFrame` loop via an uncaught exception the moment invulnerability expired) was only caught this way, not by the Python model, which has no way to know a JS constant is undefined.
  - **Character skin plumbing (minimal, one skin defined).** `drawCharacter(x, y, formT, bob, animState, skin)` reads all its fill colors from a `skin` object (`legs`/`torso`/`head`/`orb`) instead of hardcoded hex literals; `SKINS.default` is the only entry today and `getSkin()` (reads `currentSkin`) is what every call site passes. This is plumbing only, laid down because a future skin-select feature was requested — there's no selection UI and no second skin defined yet.
  - **Cavern map background (World 2)** reuses World 1's ridgeline technique (`hashCell`/`generateRidgeline`/`drawRidgeLayer` are already seed+palette parameterized, so a dark `CAVERN_LAYERS` palette reads as cave-wall silhouette with zero shared-code changes) via a small sibling `buildCavernMapBackground()`, and reuses `drawVillageNode` as-is (its cave-mouth-in-rock-mound art already reads as a cavern entrance). It deliberately skips `drawRiver`/`drawTrail` — a single-node world has no river and nothing to connect a trail between; that's intentional, not an oversight.
  - **Username + leaderboard — this project's only network dependency.** Everything else in Maze Dodge is 100% local (`localStorage` only). `SUPABASE_URL`/`SUPABASE_ANON_KEY` near the top of `game.js` are now **configured and live** (a Supabase project + `leaderboard` table + RLS policies were set up and verified end-to-end — see below); if they're ever blanked out again, every network call (`submitScore()`, `fetchLeaderboard()`) checks for both and no-ops instead of firing a request, and both wrap `fetch()` in try/catch so a real network failure degrades to "Leaderboard unavailable" rather than blocking gameplay. A username is captured once via `#usernameScreen` (shown before `#startScreen` only if `localStorage['mazeDodgeUsername']` is unset) and stored under its own key, separate from `mazeDodgeProgress`. A score is submitted (fire-and-forget, never awaited) only when `checkExitReached()` finds a new local best, reusing the same `isNewBest` check that drives the best-time banner — not on every completion. **Backend setup** (Supabase/PostgREST, chosen for a zero-SDK, zero-bundler `fetch()`-only REST API with schema-level access control): a `leaderboard` table with columns `level_index smallint check (0-6)` (**stale as of World 2**: `LEVELS` grew to 8 entries when the runner level was added at index 7, but widening this constraint to `< 8` requires the user to run an `alter table` in their own Supabase SQL editor — code alone can't do it; until then, `submitScore()` for level 7 fails silently via its existing `.catch(() => {})`, non-blocking for gameplay, same graceful-degradation pattern as a misconfigured/absent backend), `username text check (length 1-20)`, `time_seconds numeric check (0 < n < 3600)`, `created_at timestamptz default now()`; RLS enabled, with a public **select** policy (`using (true)`) and an **insert** policy mirroring the same checks, and **no** update/delete policy for the `anon` role — verified directly against the live API that inserts of valid rows succeed, invalid rows are rejected (RLS violation, not just a client-side check), and update/delete attempts silently affect zero rows. Note the project uses Supabase's newer key naming — the "Publishable key" (format `sb_publishable_...`) is what's pasted into `SUPABASE_ANON_KEY`; it's the same role/security model as the legacy JWT-style `anon` key. Known accepted limitation: no real auth, so a client could spam-insert fake rows and the same username can appear more than once — a "top runs" board, not "top players," matching the stated casual/friends-only use case.
  - **Rendering/theming**: `renderMaze()` draws per-cell floor/wall texture (`drawFloorCell`/`drawWallCell`) using a deterministic `hashCell(levelIndex, col, row)` hash (not `Math.random()`) so texture variance never flickers frame-to-frame, plus a `drawVillageGate` pulsing exit marker, collectible/dash-orb sprites, and a soft vignette — drawn fresh every frame (no offscreen cache needed; a few hundred cells/entities is trivial at 60fps). `drawCharacter(x, y, formT, bob, animState)` takes an `animState` (`{ facing, isMoving, animPhase }`, defaulting to module-level `IDLE_ANIM`) to drive the walk-cycle (leg/arm swing, left/right mirroring) in guy form and a pulse/motion-trail in orb form — `formT` still controls the guy↔orb blend independent of the animation.

When changing gameplay feel (speeds, maze layouts, hazard placements, difficulty ramp), the tunable constants are grouped near the top of `game.js`, and `LEVELS` holds all per-level content. When changing the visual theme, the palette/shape constants (`MOUNTAIN_LAYERS`, `SKY_TOP`/`SKY_HORIZON`, `GATE_COLOR`, `WALK_CYCLE_SPEED`, etc.) are grouped alongside the gameplay constants near the top of the file.
