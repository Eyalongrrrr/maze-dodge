# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Status

"Maze Dodge" — a small, self-contained 2D browser game. No git repo, no build tooling, no dependencies; plain HTML/CSS/JS with the Canvas API. Sibling project to `../2D game` ("Circle Survivor"), sharing its lightweight conventions but with unrelated gameplay logic.

## Running it

Open `index.html` directly in a browser, or serve the directory with `python3 -m http.server` and visit `localhost:8000`. A project `run` skill (`.claude/skills/run/SKILL.md`) documents a headless-Chrome/CDP recipe for driving and verifying the game end-to-end — read it before improvising a new browser-testing approach, and note its warning about launching Chrome with the target URL as the startup argument (not `about:blank` + navigate), since the latter can silently leave the tab backgrounded and `requestAnimationFrame` fully suspended.

## Architecture

- `index.html` — canvas element (800×600, no scrolling/camera — each maze fills the visible grid) plus DOM overlay divs (start/username/pause/leaderboard screens, HUD, a reusable banner). Overlays are toggled via the `.hidden` CSS class.
- `style.css` — layout and overlay styling.
- `game.js` — all game logic, in one file:
  - A `Phase` state machine (`start` → `menu` → `transform_in` → `playing` ⇄ `paused` → `transform_out` → back to `menu`) drives `update()`/`render()`. `start` is only ever visited once per session; every subsequent run loops back through `menu`.
  - The story: a traveler crossing a mountain range village by village, using a "become small and quick" trick to slip through tight places — a lighter journey framing (not a chase). Hazards (patrol/circular/bounce shapes) are the only visible obstacles — no enemy sprites, by design, just re-flavored as lanterns/beacons/wandering lights rather than pursuers. `LEVELS` holds 9 entries: 7 hand-authored World-1 maze levels named after villages along the range (Stonebridge Hollow → Skyreach Cairn, plus **Wyrmwind Col** as level 7), each an ASCII grid (`#`/`.`/`S`/`E`, 20×15 cells at `CELL_SIZE`=40) plus a `hazards` array, a one-line `flavorText`, a `collectibles` array, and (level 7 only) a `dashOrbs` array — plus World 2's two runner-mode levels (indices 7-8, "The Long Drop" and "The Narrows", see below). Levels 1–6 were generated with a recursive-backtracker maze algorithm and verified solvable via BFS before being hand-tuned with hazard placements — treat the grids as fixed data, not something to hand-edit casually without re-checking solvability. Level 7 (Wyrmwind Col) is a hand-authored single 1-wide winding corridor (see "Dash orbs" below).
  - **`WORLD_PATH`/`LEVELS` positional coupling is an undocumented-in-code invariant.** Nothing enforces it structurally, but `renderMenu()`'s node loop, `nodeIndexAt()`, and `enterLevel(idx)` all assume `WORLD_PATH[i]` is the map node for `LEVELS[i]`. Adding a level means appending to *both* arrays at the same index — never insert in the middle or reorder one without the other.
  - **Two worlds, one flat `LEVELS`/`progress`.** `WORLDS` (near `WORLD_PATH`) is a thin view layer: each entry is `{name, theme, startIndex, path}`, where `path` is that world's own node array (`WORLD_PATH` for World 1, the single-node `CAVERN_PATH` for World 2) and `startIndex` is its offset into the still-flat `LEVELS`/`progress.*` arrays — `worldIndexForLevel(levelIdx)` maps a flat index back to a world. **This is deliberately not a deeper restructuring**: `ensureProgressShape()` already generically pads `gemsCollected`/`bestTimes`/`completedLevels` to `LEVELS.length` and re-derives `unlockedIndex` from `completedLevels`, so appending World 2 as `LEVELS[7]` needed zero changes there — world-unlocking is a pure byproduct of the existing flat `unlockedIndex` gate, not a new mechanism. `nodeIndexAt(mx, my, world)` now takes a world and returns a flat level index (`world.startIndex + i`). `viewedWorldIndex` (module state) tracks which world's map is currently displayed — it follows the player's frontier by default, advances automatically the moment `finishLevel()` unlocks a new world, but otherwise stays put so browsing an earlier, fully-cleared world (e.g. to mop up missed gems) doesn't get yanked back. Small on-canvas arrow hit-targets (`worldSwitchHitTest`/`switchViewedWorld`, drawn by `drawWorldSwitchArrows`) let the player switch manually, gated by `canViewWorld()` (a world is viewable once its `startIndex <= progress.unlockedIndex`) — both go through `canvasCoords(e)` like every other click handler, never raw `clientX/clientY`.
  - **Dynamic (hazard) solvability matters, not just wall solvability.** A patrol/circular/bounce hazard placed inside a corridor that's exactly 1 cell wide with no lateral escape can make a level provably uncrossable regardless of its speed or patrol length (a continuously-moving hazard confined to a range the player must fully cross guarantees their positions coincide at some instant — an intermediate-value argument, not just "hard"). This bit two levels in practice (level 2's entry corridor and a level 4 corridor) until fixed by either opening a wall cell to create a real bypass route, or relocating the hazard into a room with genuine lateral room. When adding or moving hazards, verify a route exists that avoids the hazard's full reachable range (with margin = player radius + hazard radius), or that its reachable cells are mostly *not* 1-wide tubes — don't just eyeball it. `tools/solvability_checker.py` (stdlib-only Python, no deps) reimplements the relevant subset of `parseLevel`/`createHazard`/`updateHazard` to do this: run `python3 tools/solvability_checker.py` for a per-level, per-hazard bypass/tube report (it already knows Level 7's two hazards are intentionally dash-gated via its `DASH_GATED_HAZARDS` allowlist, so a clean run doesn't mean "no dash-gated hazards," it means none of them are *newly* broken). It also exposes `candidate_collectible_cells()` (off-route + hazard-free cell search, used to place every existing gem) and `dash_reach_ok()` (wall-clearance + sufficient-reach check for a dash past a hazard) for adding new content. Keep it in sync with `game.js` if the hazard/movement math ever changes there.
  - **`reactive` hazards need no extra solvability check.** The reachable-range argument above is spatial, not speed-dependent — a hazard's reachable range (segment/circle/rect) is fixed geometry regardless of how fast it moves; going faster only changes arrival timing, which the check never modeled. So a `reactive` flag on a hazard def (see below) is safe under the existing check as-is.
  - **Pausing**: `Escape` toggles `Phase.PLAYING` ⇄ `Phase.PAUSED` (`pauseGame`/`resumeGame`), showing/hiding `#pauseScreen`. `update()` simply has no branch for `PAUSED`, so gameplay (movement, hazards, collisions, the level timer) freezes automatically; `render()` treats `PAUSED` the same as `PLAYING` so the frozen scene stays visible under the overlay. "Exit to Map" calls the existing `returnToMenu(null)` directly — a voluntary exit never touches `progress`/`saveProgress`, unlike death or level-complete.
  - **World map (`Phase.MENU`)**: a static, once-built mountain-range vista (`buildMapBackground()` → offscreen `mapBackgroundCanvas`, blitted each frame via `drawImage`) — sky/sun/clouds, 4 seeded ridgeline layers with snow-caps (`generateRidgeline`/`drawRidgeLayer`), a river (plus a live `drawRiverFlow()` overlay for motion), a valley floor with seeded pine trees, and the trail all baked in once at startup since none of it changes at runtime. `WORLD_PATH` is a fixed array of node coordinates; `renderMenu()` draws each node as a cave entrance set into a rock mound (`drawVillageNode`) with locked (rubble-blocked, 🔒)/unlocked (dark mouth with a warm glow)/completed (gold-toned + green glow ring)/hover states — hit-testing (`nodeIndexAt`/`NODE_RADIUS`) is independent of the node art and must stay in sync with `WORLD_PATH`. A completed node also gets its best-time drawn beneath it when hovered, and the lifetime gem total is drawn top-right. Progress persists via `localStorage` under `mazeDodgeProgress` (`loadProgress`/`saveProgress`); `ensureProgressShape()` backfills newly-added fields (and pads arrays / recomputes `unlockedIndex`) onto older saves rather than discarding them. Clicking an unlocked node calls `enterLevel(index)`.
  - **The trail is a smooth curve, not a polyline.** `drawTrail(octx, path)` connects a world's nodes via Catmull-Rom-to-Bezier (`catmullRomSegments()`) rather than straight `lineTo` segments — the standard technique for a curve that passes exactly through an arbitrary point sequence, degrading gracefully to a near-straight line for a 2-node path (`CAVERN_PATH` today). Node positions themselves are untouched by this — only the connector's geometry changed, so hit-testing/art placement needed zero updates. The decorative "pebble" dots scattered along the trail sample points *on the curve* via `bezierPoint()` (explicit cubic bezier formula), not linearly between raw endpoints — if the trail math ever changes again, the pebble placement must move with it or they'll visibly float off the line.
  - **Two worlds' maps, one `renderMenu()`, sliding between them.** `renderWorldScene(world, offsetX)` draws everything that belongs to *one* world's map (background blit, river flow, nodes, frontier character, hover/flavor panel) inside a `ctx.translate(offsetX, 0)` — title text, gem total, and the switch arrows are deliberately excluded, drawn once by `renderMenu()` itself as fixed, non-sliding UI chrome. `switchViewedWorld(dir)` still flips `viewedWorldIndex` immediately (everything else in the file already treats it as "the target"), but also kicks off a **parallel, non-`Phase` transition** — `worldSwitchTimer`/`WORLD_SWITCH_TIME`/`worldSwitchFromIndex`/`worldSwitchDir` — incremented unconditionally near the top of `update()` (same pattern as `bannerTimer`). While `worldSwitchTimer < WORLD_SWITCH_TIME`, `renderMenu()` draws *both* the outgoing world (sliding off) and the incoming one (sliding in) via two `renderWorldScene()` calls at complementary offsets. This deliberately does **not** reuse the existing `TRANSFORM_IN`/`TRANSFORM_OUT` level-entry tween — that one is hard-coupled to blending one sprite between level↔menu scenes via a single `Phase` change, not a fit for cross-fading two map backgrounds while staying in `Phase.MENU` throughout. The `Phase.MENU` click handler ignores clicks entirely while a switch is mid-flight, so a rapid click can't hit a node or re-trigger another switch against stale, still-sliding positions.
  - **World-switch arrows live at the vertical edge-center (`WORLD_ARROW_Y = 300`), not a top corner — this is load-bearing, not a style choice.** A top-corner position (the original `46`) put the left arrow's hit-circle directly under `#leaderboardBtn`'s clickable box; since that DOM button has no `pointer-events: none` and sits in front of the canvas by default stacking rules, it silently swallowed clicks meant for the arrow before `worldSwitchHitTest()` ever ran. If a future world's node layout or the leaderboard button ever moves, re-check for this exact class of overlap (DOM element in front of canvas, no `pointer-events: none`) rather than assuming a canvas-space coordinate change alone is enough.
  - **Transformation**: entering/leaving a level plays a short tween (`TRANSFORM_IN`/`TRANSFORM_OUT`, `TRANSFORM_TIME`) blending the animated human silhouette into the small circular in-level form (`drawCharacter`) — purely cosmetic, not a separate animation system.
  - **In-level mechanics**: `updatePlayerMovement` does axis-separated movement against `isWallAt`/`circleCellOverlap` (wall-sliding, no corner-sticking) via a standalone `tryAxis(axis, delta)` helper (also reused by the dash mechanic), and also tracks `player.facingX/facingY/isMoving/animPhase` purely for animation. Hazards (`createHazard`/`updateHazard`) come in three types — `patrol` (ping-pongs between two points), `circular` (orbits a center), `bounce` (reflects off its own bounds rect) — and never collide with walls, so their zones must be placed inside open floor. Hazard speed scales via `speedMul * (HAZARD_SPEED_BASE + levelIndex * HAZARD_SPEED_LEVEL_INCREMENT)`, further boosted by `effectiveHazardSpeed()` when a `reactive: { triggerDist, boostMul }` field is present and the player is within `triggerDist`.
  - **Collectibles**: `collectibles: [{x,y}]` per level (empty on level 7 — a 1-wide corridor has no side branch to hide an optional pickup in). Whether a given gem is taken is read straight from `progress.gemsCollected[levelIndex][gemIndex]` (not local/runtime state) and is never reset once true — a gem, once collected, stays collected permanently across respawns, level failures, and future re-entries alike (`checkCollectiblePickups()` saves immediately on pickup, not batched to level-complete, so it banks even if the attempt is later failed). Placement must satisfy: off the maze's shortest S→E path (a genuine detour), fully outside every hazard's danger zone (both the pickup cell and the corridor leading to it — unlike the exit, an optional pickup should never require dodging a hazard), *and* fully outside the exit's own trigger radius (`exitZone.radius + PLAYER_RADIUS`, matching `checkExitReached()`'s `circlesOverlap` check) — a gem in a dead-end pocket reachable only by first passing through the exit trigger is worse than unsafe, it's flatly impossible, since `checkExitReached()` fires every frame and ends the run before a detour is possible (this shipped as a real bug on level 1 — a gem in a dead-end directly behind the gate, only reachable by walking through the exit cell itself — before being caught and fixed). `tools/solvability_checker.py`'s `candidate_collectible_cells()`/`exit_trigger_cells()` check both hazard danger zones and the exit trigger together now; re-run it after placing any new collectible.
  - **Near-miss feedback**: `checkNearMiss()` (edge-triggered per-hazard via a `nearMissArmed` flag, so it fires once per pass rather than every frame within the band) drives a short expanding/fading ring at the player (`drawNearMissFlash()`) — pure Canvas, no assets, no camera/shake system (this game has neither).
  - **Best-time**: `levelTimer` accumulates only inside `update()`'s `PLAYING` branch (so it's naturally paused-and-resumed for free, and doesn't count the `TRANSFORM_IN` tween), resets on `enterLevel()` but *not* on `respawnInLevel()` (a death mid-attempt doesn't restart the clock). Captured into `progress.bestTimes[levelIndex]` in `checkExitReached()` when it's an improvement, surfaced in the HUD (`#timeLabel`/`#bestLabel`) and on the world map.
  - **Dash orbs (level 7, "Wyrmwind Col")**: this level's grid is a single 1-wide winding corridor with **zero branches** — `parseLevel`/`isWallAt`/collision code needed no changes for this, since none of it assumes maze topology, only "is this cell a wall." A direct structural consequence: on a level with no side branches at all, *any* hazard is either dash-gated or flatly impossible (there's no way to carve a fair dodge pocket without creating an unintended shortcut between adjacent parallel runs) — this level's two hazards are both intentionally dash-gated, not an oversight. `dashOrbs: [{x,y}]` are triggered two ways: clicking one (canvas `click` listener's `Phase.PLAYING` branch, gated on both click-proximity to the orb and player-proximity via `DASH_ORB_PROXIMITY_RADIUS`), or pressing **Space** (`triggerNearestDashOrb()`, keydown handler's `phase === Phase.PLAYING` branch — no aiming needed, just triggers whichever unused orb is nearest within `DASH_ORB_PROXIMITY_RADIUS`, so you don't need to take a hand off WASD). Both call `triggerDashOrb()`: a fixed-duration (`DASH_DURATION`) burst along the player's *current facing direction* (not an authored target point — the corridor is 1-wide, so facing is inherently "along the path" at any orb), reusing `tryAxis` for wall collision (so a dash is a fast sprint, not a teleport — it still can't clip through a wall) and the existing `invulnTimer` field for its invulnerability window. Orbs are consumed **per-attempt** (rebuilt fresh in both `enterLevel()` and `respawnInLevel()`) rather than permanently — a permanently-spent orb would soft-lock a retry against the hazard it exists to bypass. Solvability reasoning for any future dash-gated hazard: the corridor up to and including the orb must pass the *ordinary* hazard-safety check with zero exceptions; only the segment immediately past the orb is allowed to fail it, and that segment instead needs (a) wall-clearance — the dash's full-length straight-line path stays on floor cells — and (b) sufficient reach — that length clears the hazard's danger zone and lands somewhere the *next* segment is ordinary-safe from.
  - **World 2 — "The Deep Cavern" runner levels (`LEVELS[7]`/`LEVELS[8]`, both `mode: 'runner'`).** A fundamentally different genre from the top-down maze engine: side-view, player fixed at `RUNNER_PLAYER_X`, spikes scroll in from the right as `runnerScroll` accumulates (`+= RUNNER_SCROLL_SPEED * dt`, the game's only "camera"/scroll concept — confined entirely to obstacle-screen-position math, nothing else gains one), jump height set by how long Space/↑ is held (charge while grounded+holding, launch on release: `velocity = RUNNER_MIN_JUMP_VEL + (RUNNER_MAX_JUMP_VEL-RUNNER_MIN_JUMP_VEL) * (chargeTime/RUNNER_MAX_CHARGE_TIME)`, then simple projectile motion under `RUNNER_GRAVITY`). A level def's `mode: 'runner'` flag is checked at exactly four dispatch points — `enterLevel()`/`respawnInLevel()` (branch to `setupRunnerLevel()` instead of `parseLevel()`), `update()` (branch to `updateRunner(dt)` instead of the maze movement/hazard/collectible/exit chain), and `render()` (branch to `renderRunnerScene()` instead of `renderMaze()` — this is a real correctness fix, not cosmetic, since both `PLAYING` and `TRANSFORM_IN` branches used to call `renderMaze()` unconditionally, which would read stale/undefined `walls`/`exitZone` for a runner level). `loseLife()` and `finishLevel()` were extracted from `checkHazardCollisions`/`checkExitReached`'s bodies so both the maze path and the new `checkRunnerCollisions`/`checkRunnerFinish` share one life-loss/level-complete implementation rather than duplicating it. Pause/resume needed zero changes — `update()` already has no `PAUSED` branch, so gameplay freezes regardless of mode. A hit **restarts the whole gauntlet from `runnerScroll = 0`**, not a checkpoint — consistent with maze respawn always resetting to the parsed `S` cell.
    Obstacles (`gauntlet: {length, obstacles: [{gaugeX, ground: {height}, ceiling?: {height}}]}`) use simple AABB overlap against the player's fixed-width bounding box; a `ceiling` field is only ever paired with a *small* `ground.height`, turning an easy-looking short hop into a precisely-timed one (too little height hits the ground spike, too much hits the ceiling spike above it) — this is the intentional "narrow gap" pattern, not every obstacle. **Solvability here isn't grid-BFS** — `tools/solvability_checker.py`'s regex requires a `grid:` key and silently skips these levels. `tools/runner_solvability_checker.py` re-simulates the exact same charge-jump physics model (**keep both files' `RUNNER_*` constants in sync**) to prove, per obstacle independently (the player chooses release timing freely, so obstacles don't interact), that some hold duration clears it, plus a spacing check between consecutive obstacles. That static model is necessarily a simplified proof of existence, not a guarantee of real in-game feel — the CDP playtest (`.claude/skills/run/SKILL.md`) is what actually exercises `updateRunner`/`checkRunnerCollisions` end-to-end; a real bug (`RUNNER_PLAYER_HEIGHT` was referenced in `checkRunnerCollisions` but never defined, silently killing the `requestAnimationFrame` loop via an uncaught exception the moment invulnerability expired) was only caught this way, not by the Python model, which has no way to know a JS constant is undefined.
  - **`LEVELS[8]`, "The Narrows" — the runner's dash orb, a *second* mechanism layered onto the same collision code.** Some obstacle combinations are **dash-gated**: when `ground.height + (ceiling.height || 0)` exceeds `RUNNER_GROUND_Y - RUNNER_PLAYER_HEIGHT - RUNNER_CEILING_Y` (146px of usable vertical room), no `runnerHeight` satisfies both constraints at once — no hold duration, however precise, clears it (this is the "tunnel" pattern: 12 tiled ground+ceiling pairs at `RUNNER_SPIKE_WIDTH` intervals, reading as one continuous seam rather than a gap-toothed row). The only way through is `runnerDashOrbs` (`{gaugeX, height}`, built fresh per-attempt in `setupRunnerLevel()` exactly like the maze's `dashOrbs`, `height` defaulting to `RUNNER_DASH_ORB_DEFAULT_HEIGHT` if omitted), triggered by **Space only** (no click support in runner mode — a fast-scrolling target isn't fair to aim at) via `triggerNearestRunnerDashOrb()`, gated on **both** horizontal proximity (`RUNNER_DASH_ORB_PROXIMITY`, scroll-distance-based: `abs(orb.gaugeX - runnerScroll)`, the runner-mode analog of the maze's pixel-distance proximity check) **and** vertical proximity (`abs(runnerHeight - orb.height) <= RUNNER_DASH_ORB_VERTICAL_TOLERANCE`) — the height gate is deliberate: standing on the ground can't reach it, so clearing the tunnel requires an actual well-charged jump (a quick tap only apexes around 33px, nowhere near the default 100px orb height) timed so its rising/falling arc passes through the height band *and* the horizontal window simultaneously. Since triggering piggybacks on the same Space key as jump-charging, the keydown handler guards with `!e.repeat` — without it, the OS's key-auto-repeat while holding Space to charge could re-fire the trigger attempt on every repeat tick at an arbitrary, platform-dependent rate, rather than requiring a genuine release-and-repress (typically: hold to charge+launch, then a separate tap once airborne at the right height). Unlike the maze's dash (a directional burst, since the player actually moves there), the runner player never moves horizontally, so triggering grants two effects instead: the timed invulnerability window (`invulnTimer = Math.max(invulnTimer, RUNNER_DASH_DURATION)` — **zero new collision code needed**, since `checkRunnerCollisions` already returns early whenever `invulnTimer > 0`), and a separate `runnerDashBoostTimer` (same duration, but a **dedicated timer, deliberately not folded into `invulnTimer`**) that multiplies `runnerScroll`'s accumulation rate by `RUNNER_DASH_SCROLL_BOOST_MUL` for a real forward-burst feel (speed-line streaks drawn in `renderRunnerScene()`, same technique as `drawRiverFlow`'s water streaks). Kept separate because `invulnTimer` is *also* the plain post-respawn grace period — folding the boost into it would speed up the scroll right at level start too, silently invalidating every obstacle's carefully-verified arrival timing. `RUNNER_DASH_DURATION` (2.4s) and `RUNNER_DASH_ORB_PROXIMITY` (40px — deliberately tight, a difficulty choice, not just "as loose as possible") are sized so that even the *earliest* possible trigger (the near edge of the proximity window) still covers the tunnel's full length with margin — this is exactly what `tools/runner_solvability_checker.py`'s `dash_group_coverage()` proves (finds the valid trigger-scroll range and its width using the *boosted* reach, not the base scroll speed, so it models the real mechanic rather than just being a conservative under-estimate), not something eyeballed. Tunnel obstacles are written as **explicit literal objects, not `Array.from(...)`-generated** — the checker's `load_runner_levels()` is a text-based parser (balanced-brace splitting, not a JS engine) and can't see runtime-computed data; this project's level-data convention (plain literals only) exists precisely so tooling never has to evaluate JS.
    **The height gate couples the orb's timing to whichever obstacle precedes it — this needs its own verification, not just "is the orb reachable in isolation."** `orb_reachable()` confirms *some* charge duration's flight arc dwells in the height band long enough to line up with the horizontal window at all, but a player also has to *start* that charge at the right scroll position, which is ~150-300px *before* the orb (charge time + rise-to-band time, converted to scroll distance) — close enough to the previous ordinary obstacle's own landing that the two aren't independent. This was caught by exhaustive grid search directly against the real `updateRunner`/`triggerNearestRunnerDashOrb` (not the Python model, which only proves per-obstacle existence, not chained feasibility) — see the CDP recipe in `.claude/skills/run/SKILL.md`: fix the earlier obstacles' known-good timings, then sweep (charge-start-time, hold-duration) for the orb jump, tapping at the first instant both proximity conditions hold, and confirm a completion exists. If a future obstacle is ever placed directly before a height-gated orb, re-run a search like this rather than trusting the per-piece checks alone.
  - **Real bug caught while wiring the dash orb, not by any automated check: the keydown handler intercepted Space unconditionally.** `window.addEventListener('keydown', ...)`'s `key === ' '` branch always called the maze's `triggerNearestDashOrb()` and `return`ed — meaning Space *never* reached `keys.add(key)`, so `updateRunner`'s `keys.has(' ')` check could never be true from a real keypress (only `arrowup` worked; the CDP tests that "verified" Level 7 originally had bypassed the real event path entirely by writing straight to the `keys` Set, which is why this went unnoticed until testing Level 8's dash trigger with genuine `Input.dispatchKeyEvent` calls). Fixed by branching on `LEVELS[currentLevelIndex].mode === 'runner'` inside that handler: runner mode calls `triggerNearestRunnerDashOrb()` *and* falls through to `keys.add(key)` (both effects of one press), while maze mode keeps its exact original return-early behavior. **Lesson for future CDP verification**: directly mutating `keys`/state via `Runtime.evaluate` is legitimate for isolating physics/collision logic, but it silently skips the real `keydown`/`keyup` listener code — at least one pass with genuine `Input.dispatchKeyEvent` is required to catch bugs that live in the event-handling wiring itself, not the simulated logic.
  - **Character skin plumbing (minimal, one skin defined).** `drawCharacter(x, y, formT, bob, animState, skin)` reads all its fill colors from a `skin` object (`legs`/`torso`/`head`/`orb`) instead of hardcoded hex literals; `SKINS.default` is the only entry today and `getSkin()` (reads `currentSkin`) is what every call site passes. This is plumbing only, laid down because a future skin-select feature was requested — there's no selection UI and no second skin defined yet.
  - **Cavern map background (World 2)** reuses World 1's ridgeline technique (`hashCell`/`generateRidgeline`/`drawRidgeLayer` are already seed+palette parameterized, so a dark `CAVERN_LAYERS` palette reads as cave-wall silhouette with zero shared-code changes) via a small sibling `buildCavernMapBackground()`, and reuses `drawVillageNode` as-is (its cave-mouth-in-rock-mound art already reads as a cavern entrance). It deliberately skips `drawRiver` — there's no cavern river, matching the theme. `drawTrail(octx, path = WORLD_PATH)` was parameterized (it originally hardcoded `WORLD_PATH`) once `CAVERN_PATH` grew a second node, so both worlds' maps now draw a connecting trail via the same function; a single-node world genuinely has nothing to connect, which is why `buildCavernMapBackground()` guards the call with `if (CAVERN_PATH.length > 1)`.
  - **Username + leaderboard — this project's only network dependency.** Everything else in Maze Dodge is 100% local (`localStorage` only). `SUPABASE_URL`/`SUPABASE_ANON_KEY` near the top of `game.js` are now **configured and live** (a Supabase project + `leaderboard` table + RLS policies were set up and verified end-to-end — see below); if they're ever blanked out again, every network call (`submitScore()`, `fetchLeaderboard()`) checks for both and no-ops instead of firing a request, and both wrap `fetch()` in try/catch so a real network failure degrades to "Leaderboard unavailable" rather than blocking gameplay. A username is captured once via `#usernameScreen` (shown before `#startScreen` only if `localStorage['mazeDodgeUsername']` is unset) and stored under its own key, separate from `mazeDodgeProgress`. A score is submitted (fire-and-forget, never awaited) only when `checkExitReached()` finds a new local best, reusing the same `isNewBest` check that drives the best-time banner — not on every completion. **Backend setup** (Supabase/PostgREST, chosen for a zero-SDK, zero-bundler `fetch()`-only REST API with schema-level access control): a `leaderboard` table with columns `level_index smallint`, `username text check (length 1-20)`, `time_seconds numeric check (0 < n < 3600)`, `created_at timestamptz default now()`; RLS enabled, with a public **select** policy (`using (true)`) and an **insert** policy mirroring the same checks, and **no** update/delete policy for the `anon` role — verified directly against the live API that inserts of valid rows succeed, invalid rows are rejected (RLS violation, not just a client-side check), and update/delete attempts silently affect zero rows.
    **Two independent gates on `level_index`, both need updating whenever `LEVELS` grows — this bit us for real when World 2 shipped.** There's the table's own `check (level_index >= 0 and level_index < N)` constraint *and* a separate condition baked into the INSERT policy's `with check` expression (`select policyname, with_check from pg_policies where tablename = 'leaderboard'` shows it) — Postgres enforces both, so widening only the table constraint still leaves inserts failing with a row-level-security error (42501), not a check-constraint error, which looks like a different bug if you don't know both exist. Fixing it needs two manual `alter` statements run by the user in their own Supabase SQL editor (code can't do either): `alter table leaderboard drop constraint <name>; alter table leaderboard add constraint ... check (level_index >= 0 and level_index < N);` and `alter policy "<policy_name>" on leaderboard with check (level_index >= 0 and level_index <= N-1 and ...same username/time_seconds checks...)`. Until both are updated for the current `LEVELS.length`, `submitScore()` for the new level(s) fails silently via its existing `.catch(() => {})` — non-blocking for gameplay, same graceful-degradation pattern as a misconfigured/absent backend. A submission that already happened during that window isn't lost, though: `progress.bestTimes[i]` is set locally regardless of network success, so once the backend is fixed, re-run `submitScore(i, username, progress.bestTimes[i])` from the browser console to send it — a fresh completion won't re-submit on its own unless it beats that already-recorded local best. Note the project uses Supabase's newer key naming — the "Publishable key" (format `sb_publishable_...`) is what's pasted into `SUPABASE_ANON_KEY`; it's the same role/security model as the legacy JWT-style `anon` key. Known accepted limitation: no real auth, so a client could spam-insert fake rows and the same username can appear more than once — a "top runs" board, not "top players," matching the stated casual/friends-only use case.
  - **Rendering/theming**: `renderMaze()` draws per-cell floor/wall texture (`drawFloorCell`/`drawWallCell`) using a deterministic `hashCell(levelIndex, col, row)` hash (not `Math.random()`) so texture variance never flickers frame-to-frame, plus a `drawVillageGate` pulsing exit marker, collectible/dash-orb sprites, and a soft vignette — drawn fresh every frame (no offscreen cache needed; a few hundred cells/entities is trivial at 60fps). `drawCharacter(x, y, formT, bob, animState)` takes an `animState` (`{ facing, isMoving, animPhase }`, defaulting to module-level `IDLE_ANIM`) to drive the walk-cycle (leg/arm swing, left/right mirroring) in guy form and a pulse/motion-trail in orb form — `formT` still controls the guy↔orb blend independent of the animation.

When changing gameplay feel (speeds, maze layouts, hazard placements, difficulty ramp), the tunable constants are grouped near the top of `game.js`, and `LEVELS` holds all per-level content. When changing the visual theme, the palette/shape constants (`MOUNTAIN_LAYERS`, `SKY_TOP`/`SKY_HORIZON`, `GATE_COLOR`, `WALK_CYCLE_SPEED`, etc.) are grouped alongside the gameplay constants near the top of the file.
